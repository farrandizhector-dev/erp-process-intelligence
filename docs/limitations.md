# Limitations — ERP Process Intelligence Platform

This document provides an honest account of the analytical and technical limitations of this project. These are not deficiencies — they are inherent constraints of the dataset, methodology, and scope. Understanding them is essential for interpreting results correctly.

---

## 1. Anonymized Dataset

The BPI Challenge 2019 dataset is fully anonymized:
- **Company IDs** (`companyID_0000` through `companyID_0003`) cannot be mapped to real subsidiary names
- **Vendor IDs** and vendor names are pseudonymized — "top violating vendor" findings cannot be acted on without the decode key
- **User IDs** (`user_000` through `user_614`) cannot be attributed to individuals or roles
- **Monetary values** use a linear translation that preserves zero and additive relationships but shifts all amounts — absolute values are not real EUR figures

**Implication:** Insights about *which specific* company or vendor drives non-compliance are illustrative, not actionable without the original data owner's decode tables.

---

## 2. Business Semantics Are Partial

The dataset covers real purchasing process data from a multinational coatings company, but the business context is incomplete:
- **No SLA contracts:** The P75 duration threshold used for SLA risk is statistical, not contractual. Real SLAs depend on payment terms, vendor agreements, and item categories that are not in the event log.
- **No cost data:** Actual EUR amounts are transformed. ROI calculations (e.g., "4,844 hours saved per month") use median wait time as a proxy for active work time — a rough estimate, not an industrial engineering measurement.
- **No organizational chart:** Handoff analysis counts resource changes but cannot distinguish intentional approvals from accidental rerouting.
- **No process documentation:** The "happy path" is computed as the most-frequent variant per flow type, not from an officially documented normative model.

---

## 3. Predictive Model Limitations

The LightGBM SLA risk classifier achieves AUC-ROC = 0.917 but has several constraints:

**Feature set is limited to case-creation attributes:**
- No event-level features (current activity, elapsed time, resources assigned) are used — these would require event-stream scoring, not batch prediction
- `vendor_freq` encodes vendor identity indirectly via case count; rare vendors are collapsed to a default of 1

**Single-year dataset:**
- Training and test data both come from 2018. Seasonal patterns learned from 2018 may not generalize to different years with different economic conditions, procurement volumes, or vendor behavior.

**Right-censoring in Q4:**
- Cases started in Oct–Dec 2018 have artificially short observed durations (the dataset window ends Dec 2019, but many cases span 100+ days). A stratified random split was used to avoid this bias, but it means the test set is not strictly temporally held-out.

**No event-time prediction:**
- The model predicts whether a case *will* exceed P75 at creation time, not *when* it will breach SLA or by how much.

---

## 4. Automation Scores Are Inferred, Not Validated

The automation opportunity scores use six heuristic dimensions (volume, batch gap, input uniformity, timing regularity, error association, wait time). These are:
- **Pattern-based:** derived from observed process behavior, not from RPA implementation data
- **Not validated in production:** no ground truth exists for "this activity was successfully automated"
- **Dependent on batch classification:** the model assumes `batch_XX` resources represent automated steps; activities they perform are used as the automation baseline

**Implication:** The scoring identifies *candidates* for automation analysis, not confirmed ROI opportunities. An RPA feasibility study would be required before implementation.

---

## 5. Single Dataset Snapshot

The event log covers approximately Jan 2018 – Dec 2019 (with the bulk of new cases starting in 2018). This means:
- **No trend analysis** beyond 13 months of data
- **No baseline comparison** — we cannot say compliance improved or worsened vs. prior years
- **Seasonal effects** cannot be distinguished from structural process differences
- **Open cases:** 21.5% of cases have no terminal activity (CR-010 fail), likely because they were still in-progress at dataset capture time, not because they were genuinely unclosed

---

## 6. Frontend Is Offline / Pre-computed

The React dashboard consumes pre-computed JSON files generated by `scripts/export_for_frontend.py`:
- **Not a live ERP connection** — data is static and reflects the state at pipeline run time
- **No write-back** — the dashboard is read-only; no actions trigger ERP changes
- **Sample-based for cases:** the Case Drilldown page shows a random sample of 1,000 cases, not all 251,734
- **Aggregated for performance:** compliance and bottleneck data are pre-aggregated; individual case-level compliance flags are not exposed in the frontend

---

## 7. Process Mining Scope

- **No Petri net discovery:** Alpha Miner / Heuristic Miner output is not used in the dashboard. Variant analysis and DFG are the primary process representations.
- **No fitness metric:** Formal conformance checking fitness (token-based replay) is not computed. Compliance rules are custom rule-based checks, not alignment-based conformance.
- **Variant explosion:** 12,660 unique variants across 251,734 cases means 99%+ of variants are rare. The "happy path" concept is fragile — 24.9% adherence to the single most-frequent variant underrepresents the many valid near-happy-paths.

---

## 8. What This Project Does Not Claim

- It does not claim the automation score rankings are definitive RPA recommendations
- It does not claim the SLA risk model would perform equally in production with real-time data
- It does not claim compliance violations are all genuine errors — some (e.g., CR-005's 21.2% "fail") reflect process variants that are business-valid
- It does not claim to be a replacement for dedicated process mining tools (Celonis, ProM, Disco)
